# 飞书 PRD → 自动化测试 · 设计文档

> 日期:2026-07-18 ｜ 承载项目:open-claude-in-chrome ｜ 状态:设计已定,待落实施计划
> 本文是 brainstorming 阶段逐节确认后的合并稿,所有[已锁定决策]均经用户逐条拍板。

## 1. 背景与目标

给**测试人员**用的自动化测试能力:喂一篇**飞书需求文档(Lark PRD)**,系统读懂需求、在真实产品上跑一遍验证、产出「通过/失败 + 截图证据」报告,并把结果**播报回飞书**。

本能力**构建在 open-claude-in-chrome 之上**——它既能读飞书文档,又能驱动任意站点浏览器自动化,天然占据"读需求 + 操作产品"两头。

自主度定位:**L2(AI 生成+执行,人审报告)**,不追求 L3/L4 全自主。

## 2. 用户与使用体验

- **使用者**:测试人员(非开发)。核心约束:**测试端零额外安装门槛**(靠一键安装解决),使用时**零授权步骤**。
- **触点**:浏览器里的 Claude 侧栏(项目已有)。
- **完整体验**:侧栏**粘一个 PRD 链接 → 点开始 → 看结果**。没有任何授权/登录步骤。
- **部署形态**:**本地,每人一套**。带来三个天然收益:
  1. 用测试人员**自己已登录的浏览器**访问被测系统 → 登录态/内网访问免搭。
  2. **对话与浏览器状态天然隔离**,人和人互不串。
  3. 无需服务器基础设施。

## 3. 范围(MVP)

**做:** 读飞书 PRD → 生成用例 → 真实产品执行 → 通过/失败+截图报告 →(人审)→ 飞书播报;共享知识库沉淀。

**明确不做(非目标):**
- 不修 bug,只发现和报告。
- 不做跨系统串测(一个任务只测一个产品/一篇 PRD)。
- 不做移动端/App,只测浏览器里的 web 页面。
- 录制重放、自愈重放(方案 C)、断点续跑 → 留后续演进。

## 5. 架构 · 组件(复用 vs 新增)

核心思路:**智能在「Agent + prompt」层,新代码只有"飞书集成 + 侧栏胶水 + 安装器",执行照用现有 4 工具。**

| 模块 | 状态 | 说明 |
|---|---|---|
| `extension/`(MV3 侧栏 + CDP 自动化) | 复用 | 仅在侧栏加"QA 工作流"入口 UI |
| `host/mcp-server.js` 4 工具(map/locate/read/act) | **复用不动** | 执行全靠它们 |
| `host/lark-adapter.js`(应用身份 docx/wiki 读、drive 传图) | 复用 | 读 PRD、传截图 |
| `host/map/*`(Pure Map 持久化) | 复用 | 定位与稳定锚点的底子 |
| 用例生成(prompt) | 🆕 | 主要是 prompt |
| 测试执行编排(prompt/workflow) | 🆕 | 薄编排层,驱动现有 4 工具 |
| Lark Base 客户端(bitable) | 🆕 | 共享知识库读写 |
| 播报器(im send_as_bot) | 🆕 | 结果发飞书群 |
| 便宜模型路由/升级 | 🆕 | 重活下沉 + 置信度升级 |
| 侧栏 QA 工作流 UI | 🆕 | 粘链接/看结果 |
| 一键安装器 | 🆕 | 打包 node+host+扩展注册+配置向导 |

### 与现有系统的冲突与规避(基于真实代码核对)

- **工具面锁 [HIGH]**:`host/mcp-tool-surface.mjs` 的 `validateClaudeToolSurface()` 硬性只允许 4 个 Pure Map 工具,其余(含所有 `lark_*`、`mcp__open-claude-in-chrome__*`)标为 FORBIDDEN。→ **新功能只能在 host/prompt 层编排,绝不给 Claude 暴露新 MCP 工具**。方案 A 天然合规。截图取证若需新能力,也放 host 层,不做成 Claude 工具。
- **侧栏并发 [MED]**:QA 长任务走 `chat-native-host`(拉起 Claude CLI),与日常聊天可能抢同一 host。→ 同一人任务**排队,一次跑一个**。落地时一并理顺 daemon/manifest 接线(见项目现存 daemon 脆弱点)。

## 6. 数据流(端到端一次任务)

```
① 触发    测试人员侧栏粘 PRD 链接 → 开跑(无授权步骤)
② 读 PRD  [应用身份] SDK 读专用知识库文档(docx.rawContent / wiki.getNode) → 需求原文
③ 生成用例 先查 Lark Base 有无既有用例(复用/增量);无则 PRD → 结构化步骤 JSON(操作+预期)
④ 定位入口 测试人员浏览器当前态 + Lark Base「功能→页面映射」找被测页(browser_map/locate)
          找不到 → 从入口导航;成功后【回写映射】(失败不回写,避免污染)
⑤ 执行判定 browser_act/locate/read 逐步驱动;每步判 通过/失败 + 截图;重活跑便宜模型
⑥ 归档    [应用身份] 每条用例跑完即写 Lark Base(bitable)+ 传截图(drive)——增量落库
⑦ 播报    [应用身份] 结果摘要(通过X/失败Y+失败详情)→ 机器人发飞书群(im)
⑧ 人审    测试人员看播报/侧栏卡片,人工确认(L2)
```

### 关键机制

- **用例库复用(③)**:同一需求二次跑复用/增量,用例库越滚越厚,省生成成本。
- **映射回写闭环(④)**:第一次找到的功能页存进 Lark Base,全团队二次直接命中——解掉"PRD 无 URL 定位难"且省 token。
- **便宜模型分层 + 升级(⑤/D10)**:机械活(按明确步骤点击、`browser_locate` 的"5选1"窄任务)下沉便宜模型;**置信度不够/失败 → 只把该步升级强模型**;歧义判定用强模型或 DOM 规则辅助。靠"Pure Map 收窄任务 + 升级兜底 + 人审"保准确,便宜模型无需 100% 准。
- **Pure Map 底座**:地图不进模型上下文(省 token);持久锚点为将来"录制重放"提供稳定性底座。

## 7. 错误处理与边界

**五条贯穿原则:**
1. 宁可停下报错,不硬跑、不瞎编。
2. 分清"真 bug"与"自动化抖动"——抖动重试 K 次,真失败留截图。
3. 拿不准就标"不确定"交人审,不硬判。
4. 绝不往共享库写脏数据(定位失败不回写错误映射)。
5. 边跑边落库,中断不丢已完成的。

| 场景 | 处理 |
|---|---|
| PRD 读不到(不在专用库/权限未配/旧版 docs) | 明确报错并指引,不硬跑(现有 adapter 已对旧版 /docs 报错) |
| 需求含糊 | 该用例标"描述不明确,待人工补充",不瞎编 |
| 功能页找不到 | 导航重试仍失败 → 标"无法定位,跳过",不阻塞其他用例,不回写错误映射 |
| 某步执行失败 | 先重试 K 次排除抖动;仍失败 → 记"失败"+截图 |
| 判定拿不准 | 标"不确定"交人审 |
| 被测系统未登录 | 检测到登录页 → 提示"请先登录被测系统",不硬闯 |
| token 过期/飞书限流 | 应用 token SDK 自动刷新;限流退避重试;写 Base 局部重试 |
| 同一人并发多任务 | 排队,一次一个 |
| 跑到一半中断 | 已完成用例已落库;MVP 先整体重跑,断点续留后续 |

## 8. 测试策略

| 测什么 | 怎么测 |
|---|---|
| 飞书集成层(读PRD/写Base/播报/传图) | 机械单元测(mock SDK),补进现有 `node --test`;可扩 `lark-simple-edit-smoke.mjs` |
| 用例生成质量 | 黄金集:代表性 PRD 定"标准用例",代码比对 |
| 便宜模型准确率 | 黄金结果集:强模型跑标准答案 → 便宜模型比对,达标才切 |
| 端到端 | 用户本人拿真实 PRD + 测试环境手验 |
| 不破坏现有 | 新模块不踩坏现有 147 回归,不动 4 工具面 |

原则:机械的用代码测;便宜模型/用例生成靠黄金集度量;端到端用户自验。不堆无用测试脚手架。

## 9. 前置准备(管理员一次性)

1. 飞书开发者后台建自建应用「自动化测试」(已建),加**机器人**能力。
2. 权限(tenant 单边,批量导入,发布后生效):
   ```json
   { "scopes": { "tenant": [
     "wiki:wiki:readonly", "docx:document:readonly",
     "bitable:app", "im:message:send_as_bot", "drive:drive"
   ] } }
   ```
   (scope 串名以权限列表为准)
3. 建**专用知识库**(如"待测需求库"),**授权给应用**;要求 PM/测试把 PRD 放进该库。
4. 建一个 **Lark Base**(多维表格),建表:用例库 / 执行记录 / 功能→页面映射;拿到 `app_token` 配到本地 host。
5. 建一个"测试结果"飞书群,把机器人拉进去(播报目标)。

## 10. 风险与未决项

- **[待定] 播报目标**:MVP 建议发固定"测试结果群"(chat_id);按人单发需拿 open_id,较麻烦,暂不做。
- **[风险] 便宜模型准确率 / 判定飘**:靠分层+升级+人审压低,非消灭;需黄金集持续度量。
- **[风险] 侧栏 daemon/manifest 接线现存脆弱**,落地时一并理顺。
- **[约束] PRD 必须放进专用知识库**,否则应用读不到。
