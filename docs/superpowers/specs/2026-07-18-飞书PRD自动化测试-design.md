# 飞书 PRD → 自动化测试 · 设计文档

> 日期:2026-07-18(v2,含评审修订)｜ 承载项目:open-claude-in-chrome ｜ 状态:设计已定,待落实施计划
> 本文为 brainstorming 逐节确认稿。v2 相对 v1 的重大变化:执行从"AI 每步现场驱动"改为"**AI 生成锚点化脚本 → 确定性重放**",并修正了对现有代码的错误"复用"表述。

## 1. 背景与目标

给**测试人员**用的自动化测试能力:喂一篇**规范化的飞书提测文档(Lark PRD)**,系统读懂需求、生成可重放的测试脚本、在真实产品上执行、产出「通过/失败 + 截图证据」报告,并**播报回飞书**。

构建在 open-claude-in-chrome 之上:它具备浏览器自动化底子(CDP + Pure Map 定位)。**注意:飞书文档的"应用身份 SDK 读取"当前并未接线,需新建/复活(见 §5)。**

自主度定位:**L2(AI 生成+执行,人审报告)**。

## 2. 用户与使用体验

- **使用者**:测试人员(非开发)。核心约束:测试端**零额外安装门槛**(靠一键安装)、使用时**零授权步骤**。
- **触点**:浏览器里的 Claude 侧栏(项目已有),作为**触发器 + 结果展示 UI**。
- **完整体验**:侧栏**粘一个提测文档链接 → 点开始 → 看结果**。无授权/登录步骤。
- **部署形态**:**本地,每人一套**。收益:①用测试人员自己已登录浏览器访问被测系统(登录态/内网免搭);②对话与浏览器状态天然隔离;③无需服务器。

## 3. 范围(MVP)

**做:** 规范提测文档 → 生成用例/脚本 → 真实执行(首跑探路→固化脚本;回归确定性重放)→ 通过/失败+截图报告 →(人审)→ 飞书播报;共享知识库沉淀。

**明确不做(非目标):**
- 不修 bug,只发现和报告。
- 不做跨系统串测(一个任务只测一个产品/一篇文档)。
- 不做移动端/App,只测浏览器 web 页面。
- 断点续跑留后续。

## 5. 架构 · 组件(复用 vs 新增)

核心:**host 侧编排器**是大脑,读规范文档→生成锚点化脚本→驱动浏览器→判定→写库→播报。Claude 侧栏只负责触发和展示。

### 现有代码的真实状态(经硬证据核对,修正 v1 错误)

- ✅ **可复用**:`host/mcp` 底层的浏览器自动化(Pure Map 建图/定位/读取/操作)、`host/map/*` 持久化、extension 的 CDP 能力。
- ✅ **SDK 存在但非声明依赖**:`node_modules/@larksuiteoapi` v1.70.0 在,但 `host/package.json` 未声明 → **必须补进依赖**,否则 npm install 不带来。
- ⚠️ **应用身份 SDK 适配器当前未接线**:`host/lark-adapter.js`(SDK 应用身份读 docx / 传图)**当前工作树已删除**,仅存于 git HEAD(commit 4b08c06)→ **需从历史复活或重建**,不是"直接复用"。
- ⚠️ **现有活着的 Lark 读取是 DOM 深读**(`extension/lark-deep-reader.js` + `host/lark-index/*` + `host/lark-reader/*`),靠测试人员浏览器登录态爬当前页面 —— 与"应用身份 SDK 读专用知识库"是**两条不同路径**。本方案读 PRD 走 **SDK 应用身份**;DOM 深读不用于读 PRD(避免两套逻辑,除非作为降级 fallback,由实施阶段定)。

### 与现有系统的冲突与规避(硬证据核对)

- **工具面锁 [已规避]**:`host/mcp-tool-surface.mjs` 的 `validateClaudeToolSurface()` 只允许 4 个 Pure Map 工具暴露给 Claude,其余全 FORBIDDEN。→ 本方案**编排器在 host 侧直接调底层库函数,不经 Claude 的 MCP 工具调用**(D11),从根上绕开该锁;也**不给 Claude 新增工具**。
- **侧栏并发 [MED]**:同一人 QA 任务与日常聊天可能抢 host。→ 任务排队,一次一个;落地时理顺现存 daemon/manifest 脆弱点。
- **底层库调用面待核实**:编排器"直接调 map/locate 库函数"需在实施前核实这些能力是否有可被 host 内部直接调用的接口(而非只有 MCP 入口)。

## 6. 数据流(端到端一次任务)

```
① 触发    测试人员侧栏粘提测文档链接 → 编排器开跑(无授权步骤)
② 读文档  [应用身份·SDK] 读专用知识库文档(docx.rawContent / wiki.getNode) → 需求原文 + 入口信息
③ 出脚本  查 Lark Base 有无该需求既有脚本/用例(复用/增量);
          无则 规范文档 → 生成「锚点化步骤序列」(操作+预期+锚点)  ← 可跑便宜模型
④ 定入口  文档写了入口 → 用之;没写 → 查 Lark Base「功能→页面映射」;
          仍无(冷启动)→ 从 base URL 首跑探路 / 提示人补
⑤ 执行    重放脚本:锚点命中直接操作(不调模型);
          锚点失效 → 自愈(便宜模型现场 locate 找到→继续→更新脚本锚点)  ← 自愈才调模型
          每步判 通过/失败;不确定→标记交人审;截图取证
⑥ 固化    首跑成功的脚本 + 新学到的「功能→页面映射」写回 Lark Base(自检通过才入库)
⑦ 归档    [应用身份] 每条用例结果 + 截图 增量写 Lark Base(bitable + drive)
⑧ 播报    [应用身份] 结果摘要 → 机器人发飞书群(im)
⑨ 人审    测试人员看播报/侧栏卡片确认(L2)
```

### 关键机制

- **规范文档为一等输入(D1/D2)**:入口、范围写进文档,是脚本可靠生成的前提(对齐主流"入口是已知输入")。
- **脚本复用 + 映射沉淀(③④⑥)**:同需求二次跑复用脚本;功能→页面映射越滚越准,解"文档没写入口"。
- **确定性重放为稳态(D9/D10)**:回归几乎不调模型 → 省 token + 稳定,对齐主流脚本化。
- **锚点 + 自愈(D12/⑤)**:脚本绑 Pure Map 持久锚点(比录坐标稳);锚点失效自动用便宜模型 locate 补并更新锚点,非直接判失败。
- **便宜模型只在窄任务点用**:生成、首跑探路、自愈——且 locate 是"5选1"窄任务;不确定升级强模型;人审兜底。便宜模型无需 100% 准。

## 7. 错误处理与边界

**五条贯穿原则:** ①宁可停下报错不硬跑不瞎编 ②分清真 bug 与自动化抖动(抖动重试 K 次) ③拿不准标"不确定"交人审 ④绝不写脏数据(定位/自愈失败不回写错误锚点/映射) ⑤边跑边落库。

| 场景 | 处理 |
|---|---|
| 文档读不到(不在专用库/权限未配/旧版 docs) | 明确报错并指引,不硬跑 |
| 文档不规范(缺入口/预期含糊) | 缺入口→查库→仍无则提示补;含糊用例标"待人工补充",不瞎编 |
| 功能页找不到(冷启动) | 探路重试仍失败→标"无法定位,跳过",不阻塞其他用例,不回写错误映射 |
| 某步执行失败 | 重试 K 次排除抖动;仍失败→记"失败"+截图 |
| **Pure Map 锚点失效(页面改版→可能静默误判)** | ⭐ 重放时锚点疑似失效**不直接判通过/失败**,先自愈重新 locate 核对;`browser_map refresh=true`("仅限用户明确要求")在此定义为"测试任务触发即视为允许重建",避免拿旧图误判 |
| 判定拿不准 | 标"不确定"交人审 |
| 被测系统未登录 | 检测到登录页→提示"请先登录被测系统",不硬闯 |
| Lark Base 写入失败(表结构变/被删/字段改名) | 校验表结构;失败→局部重试+明确报错,不静默丢结果 |
| 生成的脚本 JSON 解析失败 | 该用例标"脚本生成异常",重生成一次或交人审,不带病执行 |
| 截图上传 drive 失败 | 重试;仍失败→报告标"证据缺失"但保留判定 |
| token 过期 / 飞书限流 | 应用 token SDK 自动刷新;限流退避重试 |
| 同一人并发多任务 | 排队,一次一个 |
| 跑到一半中断 | 已完成用例已落库;MVP 先整体重跑,断点续留后续 |

## 8. 测试策略

| 测什么 | 怎么测 |
|---|---|
| 飞书集成层(读文档/写Base/播报/传图) | 机械单元测(mock SDK),补进现有 `node --test` |
| 脚本生成质量 | 黄金集:代表性规范文档定"标准脚本/用例",代码比对 |
| 便宜模型准确率(生成/自愈) | 黄金结果集:强模型跑标准答案 → 便宜模型比对,达标才切 |
| 重放稳定性 | 同一脚本对同一页面重放多次,核一致性;人为改页面验自愈 |
| 端到端 | 用户本人拿真实规范文档 + 测试环境手验 |
| 不破坏现有 | 新模块不踩坏现有全部回归(以最新跑通数为准),不动 4 工具面 |

原则:机械的用代码测;生成/自愈靠黄金集度量;端到端用户自验。不堆无用测试脚手架。

## 9. 前置准备(管理员一次性)

1. 自建应用「自动化测试」(已建),加**机器人**能力。
2. 权限(tenant 单边,批量导入,发布后生效):
   ```json
   { "scopes": { "tenant": [
     "wiki:wiki:readonly", "docx:document:readonly",
     "bitable:app", "im:message:send_as_bot", "drive:drive"
   ] } }
   ```
   (串名以权限列表为准)
3. 建**专用知识库**(如"待测需求库")并授权给应用;要求 PM/测试把提测文档放进去。
4. **定义提测文档规范模板**(必含:被测系统入口/URL、测试范围、每条需求的预期)。
5. 建 **Lark Base**:用例/脚本库、执行记录、功能→页面映射;`app_token` 配到本地 host。
6. 建"测试结果"飞书群,拉入机器人(播报目标)。

## 10. 待实施前必须定清的未决项(评审提出)

- **[未决] 用例/脚本 JSON schema**:字段结构(操作类型、锚点、预期、判定方式)。是 §8 黄金集比对的对象。
- **[未决] 用例/脚本复用匹配 key**:用文档 ID?需求条目?文本相似度?谁判定复用。
- **[未决]「功能→页面映射」存储 key + 匹配算法**:key 用稳定页面身份(如 `page-key.mjs` 的 `lark:<ws>:<token>` 思路对应到被测系统)还是功能名;自然语言功能描述如何匹配到 key(精确/语义/LLM)。
- **[未决] 编排器调用底层能力的接口**:确认 map/locate 有可供 host 内部直接调用的 API(非仅 MCP 入口)。
- **[待定] 播报目标**:MVP 建议固定"测试结果群"(chat_id);按人单发需 open_id,暂不做。
