# 飞书 PRD → 自动化测试 · 设计文档

> 日期:2026-07-18(v4,含三轮评审 + 一轮对抗红队)｜ 承载项目:open-claude-in-chrome ｜ 状态:设计已定,待用户复核 → 实施计划
> 执行=**AI 读规范文档 → 生成锚点化脚本 → 确定性重放**。v4 并入对抗红队(查飞书官方文档 + grep 代码)的加固:D11 解耦程度校正、自愈语义误判防护、raw_content 表格陷阱、飞书授权真空、新增「结果可信度」节。

## 1. 背景与目标

给**测试人员**用:喂一篇**规范化飞书提测文档**,系统读懂需求、生成可重放脚本、真实产品执行、产出「通过/失败 + 证据」报告,并**播报回飞书**。构建在 open-claude-in-chrome 之上(CDP + Pure Map 定位)。自主度 **L2(AI 生成+执行,人审报告)**。

## 2. 用户与使用体验

- **使用者**:测试人员(非开发)。约束:测试端**零额外安装门槛**、使用时**零授权步骤**。
- **触点**:浏览器 Claude 侧栏,作触发器 + 结果展示。
- **体验**:侧栏**粘提测文档链接 → 点开始 → 看结果**。
- **部署**:本地每人一套。收益:①用自己已登录浏览器访问被测系统;②对话/状态天然隔离;③无服务器。

## 3. 范围(MVP)

**做:** 规范提测文档 → 生成用例/脚本 → 真实执行(首跑探路→固化;回归确定性重放)→ 通过/失败+证据报告 →(人审)→ 飞书播报;共享库沉淀。
**不做:** 不修 bug(只发现报告);不跨系统串测;不测移动端/App;断点续跑留后续。

> ⚠️ **范围提示**:v2 起为绕工具面锁,执行换成"脚本生成 + 锚点绑定 + 确定性重放 + 自愈"整套新引擎(host 编排器/生成器/重放器/自愈全 🆕),**工作量明显重于最初"薄编排复用 4 工具"的设想**,立项排期按此估。

## 5. 架构 · 组件(复用 vs 新增)

核心:**host 侧编排器**读规范文档→生成锚点脚本→驱动浏览器→判定→写库→播报;Claude 侧栏只触发/展示。

### 现有代码真实状态(硬证据核对)

- ✅ **可复用**:`host/map/map-engine.mjs` 的 `MapEngine.locate` 是**纯本地函数**(只查已持久化 map,无浏览器往返)→ 编排器可直接 import 调用。
- ⚠️ **`map/read/act` 不是纯库调用**:`handleBrowserMap/Read/Act`(`host/mcp-server.js`)最终写到**唯一的全局 `nativeHostSocket`**(单例,新连接销毁旧连接)→ 经 `native-host.js` TCP → Chrome 原生消息 + `chrome.debugger`(CDP,会弹"正被自动化软件控制"横幅)。编排器要么塞进 mcp-server 同进程,要么复用其 TCP 握手协议共享该单例通道。**工作量远大于"直接调库"字面**。
- ✅ **SDK 存在但非声明依赖**:`node_modules/@larksuiteoapi` v1.70.0 在,`host/package.json` 未声明 → **必须补进依赖**。
- ⚠️ **飞书 SDK 适配器须全新重建**:旧 `host/lark-adapter.js` 及依赖(`capability-registry`/`chat-native-relay`/`lark-session-backend`)已被提交 `4792012 回退 daemon` 从 HEAD 删除(属已放弃的旧 daemon 架构)→ 照当前 v0.15.3 架构**从零重建轻量 SDK 客户端**,别 `git checkout` 旧文件。
- ⚠️ **现有 Lark 读取是 DOM 深读**(`extension/lark-deep-reader.js` 等),与"应用身份 SDK 读专用知识库"是两条路;本方案读 PRD 走 SDK 应用身份。

### 冲突与规避

- **工具面锁 [已规避]**:`mcp-tool-surface.mjs` 的 `validateClaudeToolSurface()` 只校验**暴露给 Claude** 的工具列表,不管 host 内部怎么调 → 编排器在 host 侧运行、不给 Claude 加工具,从根上绕开。
- **单例通道并发 [HIGH·待验证]**:`map/read/act` 共享单例 socket + 每 tab 独占 `chrome.debugger`,且与侧栏会话共用 → 不只是"业务排队",须在实施前验证:①编排器与 mcp-server 进程关系;②单例通道能否并发驱动不同 tab 不互相超时;③CDP 横幅对"零授权体验"的影响。(见 §11)

## 6. 数据流(端到端一次任务)

```
① 触发    侧栏粘提测文档链接 → 编排器开跑(无授权步骤)
② 读文档  [应用身份·SDK] 读专用知识库文档 → 需求原文 + 入口
          ★ 先探测有无 table 块:有表格 → 走 blocks 接口结构化抽取或标"表格需求待人工确认边界",
            不无脑吃 raw_content(raw_content 会把表格打平/丢图,而 PRD 关键常是验收表格)
③ 出脚本  查 Lark Base 有无既有脚本(复用/增量);无则 规范文档 → 锚点化步骤序列  ← 可便宜模型
④ 定入口  文档写了→用之;没写→查 Lark Base 映射库;仍无→base URL 探路 / 提示补
⑤ 执行    重放:锚点命中直接操作(不调模型)
          锚点失效 → 先【分类失败原因】(selector / timing / data / rendering):
              · 仅 selector 类 → 自愈(便宜模型 locate);自愈后做【语义一致性二次校验】
                (比新旧锚点父子/上下文/role 是否一致,不只匹配分),校验过才继续并更新锚点
              · 其余类(timing/data/rendering)→ 直接标"不确定",不硬当锚点问题
          ★ 任何刚发生过自愈的用例,无论后续通过与否,默认打【高优先级人审】标记
          每步判 通过/失败/不确定;截图取证(见 §8 按需详略)
⑥ 固化    首跑成功脚本 + 新学映射写回 Lark Base(自检过才入库);
          回归中自愈成功也【无论首跑与否都更新锚点回写】,否则改版页每次回归反复自愈,D10 稳态失效
⑦ 归档    [应用身份] 每条结果 + 证据 增量写 Lark Base(bitable + drive)
⑧ 播报    [应用身份] 结果摘要 → 机器人发飞书群(im)
⑨ 人审    测试人员看播报/侧栏卡片确认(L2)
```

## 7. 错误处理与边界

**五条原则:** ①宁停下报错不硬跑不瞎编 ②分清真 bug 与自动化抖动(抖动重试 K 次) ③拿不准标"不确定"交人审 ④绝不写脏数据 ⑤边跑边落库。

| 场景 | 处理 |
|---|---|
| 文档读不到 / 不规范(缺入口) | 报错指引;缺入口→查映射库→仍无提示补;含糊用例标"待人工补充" |
| 表格型需求(raw_content 打平) | 探测到 table 块走 blocks 抽取或标"待人工确认边界"(§6②) |
| 功能页找不到(冷启动) | 探路重试仍失败→标"无法定位,跳过",不阻塞、不回写错误映射 |
| 某步执行失败 | 重试 K 次排抖动;仍失败→记"失败"+截图 |
| **锚点失效(页面改版→静默误判)** | ⭐ 先分类失败原因;仅 selector 类才自愈;自愈后**语义一致性二次校验**;**刚自愈过的用例默认高优先级人审**;编排器直接重建地图核对(不经 MCP `refresh` 参数校验,那约束只作用于侧栏对话场景) |
| 便宜模型自评过自信 | 升级强模型不只信自评,叠加**历史误判率独立阈值**(D10) |
| 判定拿不准 | 标"不确定"交人审 |
| 被测系统未登录 | 检测到登录页→提示"请先登录",不硬闯 |
| Lark Base 写入失败(表结构变/被删/字段改名) | 校验表结构;局部重试+明确报错,不静默丢结果 |
| 脚本 JSON 解析失败 | 标"脚本生成异常",重生成一次或交人审,不带病执行 |
| 截图上传 drive 失败 | 重试;仍失败标"证据缺失"但保留判定 |
| token 过期 / 飞书限流 | 应用 token SDK 自动刷新;限流退避重试 |
| 同一人并发多任务 | 排队,一次一个(叠加 §5 单例通道限制) |
| 跑到一半中断 | 已完成用例已落库;MVP 先整体重跑 |

## 8. 结果可信度(让人信服 · 考虑测试时间)

**一条原则:证据不对称——通过=轻,失败/不确定=重。** 人只需细看没过的;截图只在关键点,不每步;报告默认摘要、可展开。

| 机制 | 做法(省时/不啰嗦) |
|---|---|
| **判定三态** | 通过 / 失败 / **不确定(需人工)**;拿不准不硬判 |
| **证据按需详略** | **只在断言点和失败点截图**;通过=一行结论(+最终态 1 图);失败/不确定=期望 vs 实际 + 前后截图 + 那几步 trace |
| **通过也留最小痕迹** | 每条判定记:用了哪个锚点、是否发生过自愈、便宜模型还是强模型、置信度——供复核,不盲信"通过" |
| **报告默认摘要可展开** | 顶部「通过X/失败Y/不确定Z + 覆盖 M/N 条需求」;仅失败/不确定项展开,过的折叠 |
| **可复现** | 锚点脚本确定性;附"一键重放这条";⚠️ 自愈路径含 LLM 非确定性,刚自愈过的用例可信度另标 |
| **需求覆盖率溯源** | 用例 JSON 强制含"对应 PRD 哪条需求"字段(§11);报告给覆盖率,防"悄悄漏测某条验收标准还全绿" |
| **长期度量(异步,不占测试时间)** | 生产期抽样人工复核"通过"结果统计**误判率**;记"AI 判定 vs 人工改判率";改判率长期偏高=AI 层未到托付门槛。回写 Lark Base,驱动便宜模型达标切换 |

## 10. 前置准备(管理员一次性)

1. 自建应用「自动化测试」(已建),加**机器人**能力。
2. 权限(tenant 单边,批量导入,发布后生效):
   ```json
   { "scopes": { "tenant": [
     "wiki:wiki:readonly", "docx:document:readonly",
     "bitable:app", "im:message:send_as_bot", "drive:drive"

   ```

6. 建"测试结果"飞书群,拉入机器人;取其 chat_id 配到 host。
